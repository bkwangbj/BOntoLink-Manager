# 协作术语备忘

> 记录与 Claude(或任何工程师)协作时,表述可以更专业/精准的地方。
> 目标:描述 UI 用「组件名 + 部位名」;描述需求给「现状 → 期望 → 触发条件」;业务上分清「对象类型 / 属性 / 看板 vs 图表」。

---

## 一、前端组件与交互术语

| 口语原话 | 更准确的说法 | 可直接套用的例句 |
|---|---|---|
| “燕尾”弹出浮窗 | 浮层的**指示箭头(arrow / caret)**;“紧着在行下面”=**紧贴该行下方、左对齐** | 「筛选浮层的**指示箭头**对准属性行,**浮层紧贴该行下方弹出**。」 |
| 下**来**宽度和上面一致 | “下来”=**下拉(dropdown)**;“上面”=**触发按钮** | 「**下拉浮层宽度与触发按钮一致**。」 |
| **行高**一致 | 这里其实是**菜单项高度(item height)** | 「**菜单项高度、字号、图标**和『设计/预览』下拉统一。」 |
| 弹框 / 侧边抽屉(说得不错,巩固) | **模态弹框(modal)** vs **侧边抽屉(drawer)**;可放大=**最大化**;可拉伸=**可调宽(resizable)** | 「用**侧边抽屉**,支持**拖拽调宽 + 最大化**。」 |
| “新建图标+新建” | **新增一个带图标的『新建』菜单项** | 「下拉底部**新增菜单项『新建』,前面配图标**。」 |

## 二、数据可视化 / 看板术语

| 口语原话 | 更准确的说法 | 例句 |
|---|---|---|
| “宽度适配问题” | **画布宽度未自适应容器(未铺满)**;行业词:**流式/自适应宽度(fluid / adaptive width)** | 「默认看板**画布宽度没自适应容器**,右侧留白,要**铺满整宽**。」 |
| “图表 / 看板”混用 | 整块=**看板 / 仪表盘(dashboard)**;单个=**图表(chart)** | 「**新建看板**时打开**空白画布**,弹框标题叫『新建看板』。」 |
| “这些参数”(指弹框字段) | **表单字段(form fields)**,不是“参数” | 「复用保存弹框的**这几个表单字段(名称/描述/可见性)**,只改**弹框标题**。」 |
| “单独走一套” | **与另一模式解耦,复用组件自带流程** | 「看板的保存/新建**走设计器自带那套**,和列表模式**各自独立**。」 |
| “空白设计页面” | **空白画布 + 设计模式(design mode)** | 「新建看板**打开空白画布并进入设计模式**。」 |

## 三、业务 / 数据模型术语(最容易误解)

| 口语原话 | 更准确的说法 | 说明 |
|---|---|---|
| “属性对象类 / 普通对象类” | **普通属性(标量)** vs **对象属性 / 关联属性(指向其他对象类型)** | “对象类”在本体系统里=对象类型/Class,另有含义,混用会理解偏 |
| “全量下切换到看板” | “全量”=**默认列表(全部数据的列表视图)** | 「在**默认列表视图**下切到看板时……」 |

## 四、需求表达方式(影响协作效率)

**1. 过于笼统**
- 原话:「调整这里的交互」「是不是对应不上了,逻辑混乱了」
- 更好:**现状 + 期望 + 触发条件** 三段式
- 例句:「**现状**:切到看板后下拉混进了列表模式的设计;**期望**:看板模式只列看板设计;**触发**:点看板页签时。」

**2. 只说“不对”,不说“应该怎样”**
- 原话:「而不是我新建编辑后的看板啊」
- 更好:补上**期望行为**
- 例句:「切走再切回看板,应**恢复我上次选中的那个看板**,而非回到默认看板。」

---

## 五、常见错别字
- 下**来** → 下**拉**;UI **开**起来 → UI **看**起来;紧**着**在 → 紧**贴**。

---

*用法:提需求前对照第四节的三段式;涉及 UI 时用第一/二节的部件名;涉及数据模型时用第三节的正名。*
